## 한 줄 요약

기술·비용·보안 모두 공유 가능한 상태였으나, **개인용으로 쓰기로 최종 결정.** 코드는 손대지 않음.

## 검토 결과 (10명 이내 학생 공유 가정)

| 항목 | 결과 | 근거 |
|---|---|---|
| 기술적으로 공유 가능? | ✅ 가능 | Supabase 클라우드 백엔드 + 로그인 구조라 학생마다 각자 계정·각자 데이터로 굴러감 |
| 무료(Supabase Free)로 되나? | ✅ 충분 | 무료 한도: 월 활성유저 5만 명 / DB 500MB / 파일 1GB / 트래픽 5GB. 10명 + 텍스트만 쓰면 여유 |
| 사진 첨부 용량 걱정? | ✅ 해당 없음 | 사진 첨부 안 쓰기로 함 → 1GB 스토리지 무관 |
| 보안 — 학생끼리 일기 안 보이나? | ✅ 안전 | 아래 "보안 점검" 참조 |

### 무료 플랜 운영상 주의 (참고용, 사고 아님)
- **1주일 미사용 시 프로젝트 자동 일시정지**: 아무도 1주일간 안 쓰면 DB가 잠듦. 데이터는 안 날아가고, 대시보드에서 다시 켜면 됨. (연수 끝나고 다 같이 방치하면 발생)

---

## 보안 점검 (실제 침투 테스트로 확인)

앱에 박혀있는 공개키(`sb_publishable_...`, anon/publishable 키 — 노출돼도 안전한 종류)로
"바깥에서 해커처럼 찔러보는" 테스트를 직접 수행함. 결과:

| 시도한 공격 | 결과 | 판정 |
|---|---|---|
| 로그인 없이 `user_data` 전체 읽기 | `[]` (0건) | 🟢 차단 |
| 로그인 없이 데이터 쓰기(INSERT) | 거부 (HTTP 400) | 🟢 차단 |
| 전체 행 개수(count) 캐내기 | `0` | 🟢 차단 |
| 숨은 다른 테이블 뒤지기(profiles/diaries 등) | 전부 404 | 🟢 노출 없음 |

→ **RLS(Row Level Security)가 제대로 켜져 있음.** 공개키로는 로그인한 본인 데이터만 접근 가능.
   학생들이 각자 가입해도 서로 일기를 볼 수 없음 = "앱만 똑같이, 각자 데이터" 구조가 이미 완성돼 있음.

## 만약 나중에 공유한다면 (당시 검토한 추가 작업 — 지금은 안 함)

- **가입 잠그기(초대코드 방식)**: 앱이 열자마자 익명 자동가입되는 구조라, 주소가 새어나가면 모르는 사람도
  자동가입됨. 막으려면 진입 화면에 초대코드(예: `BAGUIO2026`) 입력 단계를 추가하면 됨.
  - 수정 위치: `src/BaguioApp.jsx`의 `initSync()` 호출 앞 + 코드 입력 컴포넌트 1개 추가
  - 성격: 강한 보안이 아니라 "낯선 사람 막는 문턱". 진짜 데이터 보안은 이미 RLS가 담당.
  - **이번엔 개인용 유지로 결정해서 구현하지 않음.**
